Build Your Own Engine (II) - UI Redesign

Created2026-03-28|Updated2026-03-28|Technology SharingEngine Related
|Post Views:

Written before

The previous article covered the engine’s initial architecture, basic rendering, physics simulation, and parallelization. I was planning to wrap it up and focus on my thesis and spring recruitment, but one day I stumbled upon Krystallos Engine from another developer. Something just clicked—I couldn’t sleep, and after a week of grinding, I got this major UI overhaul done.

Project Management

Written before

The engine now supports multiple projects, so the first step when opening the engine is to select a project. The project management interface is shown at startup, allowing you to create new or open existing projects.

Startup Interface

At startup, check if projects exist—if not, show the project selector:

Project Selector

The project selector displays all available projects, supporting create, open, and rename operations:

Written behind

Project management involves project path resolution, JSON read/write, and automatic scene loading. It took some doing, but it finally works.

Docking System

Written before

After selecting a project, you enter the main interface. The previous UI used a primitive “manual region division” approach: Hierarchy occupied the left 1/8, Scene the middle 1/2, and Inspector the right 1/4. While functional, it had terrible extensibility—no drag-and-drop, no resizing, and no multi-view layouts like Unity. Fortunately, ImGui has built-in support for Docking, so that’s what I implemented.

Enabling Docking

When initializing ImGui, enable Docking and Viewport for multi-window support:

Setting Up DockSpace

The core of the docking system is creating a fullscreen DockSpace where all child windows can be docked:

Initial Layout

The first time running, you need to manually divide window regions. The classic five-part layout: left top/bottom (Hierarchy + Project), center top/bottom (Scene + Game), right (Inspector):

Layout Management

The engine supports saving and loading UI layouts using ImGui’s built-in INI mechanism:

Written behind

The docking system advanced the engine’s UI from the “stone age” to a modern editor baseline. Now you can freely drag windows, resize them, and save layouts. However, the tradeoff is that ImGui’s INI file becomes important—once corrupted, the UI becomes a mess.

File Browser

Written before

After selecting a project, you’ll see the Project window. The previous Project window was basically just an image viewer with no folder navigation. This time I completely rewrote it into a complete file browser.

Basic Structure

The file browser maintains current path, expanded folders list, file icon mappings, and more:

Drawing Flow

Uses a left-right split layout: left shows folder tree, right shows file list:

Folder Tree Display

Left side shows tree structure with expand/collapse:

File List Display

Right side shows files in current folder with icons based on extension, supports single/double click:

Right-click Menu and Drag

Right-click menu supports create script/folder/scene, rename, delete, etc. For C# scripts, also supports drag to Inspector window to auto-add components:

Script Creation

Right-click menu can create new C# scripts with auto-generated template code:

Written behind

Now the Project window finally feels like a modern editor’s file browser. But the code really ballooned—from a few hundred lines to nearly a thousand.

Inspector Panel

Written before

After clicking an object in the Scene view, Inspector shows all component properties. This time I added a useful feature: 3D Model Preview.

Rendering Target

Model preview needs a separate Framebuffer with 256x256 resolution:

Preview Rendering

Display in Inspector

3D model preview greatly improves the experience—at least you know what the model looks like without having to check the Scene view.

Script System

Written before

I tried C++ scripts before, but manually registering all variables and functions was too tedious. The DLL approach requires compiling like UE, which isn’t great for UX. So I went with the C# route like Unity.

C# Base Library

The engine provides a base C# library DittoEngine.dll, similar to Unity’s UnityEngine:

Field Parsing

Public variables in C# scripts need to be parsed on the C++ side for Inspector display and editing:

Dynamic Compilation

Compile C# to DLL at runtime:

Inspector Display

Show script’s public fields in Inspector with real-time editing:

Written behind

The script system currently only supports parsing public fields. The C# side’s Start/Update aren’t really connected to C++ yet. To truly implement runtime script invocation like Unity, I need to integrate the Mono VM or Dotnet runtime—added to the Todo list.

Written behind

That’s it—the engine UI has been completely overhauled. Project management, docking system, file browser, model preview, script system—these are ordinary features in commercial engines, but I’ve put in the work for a personal engine. Next, I need to survive the thesis and spring recruitment, keep grinding, or I’ll never graduate or find a job… might as well live under a bridge (
